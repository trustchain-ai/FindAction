# FindAction · v0.1 · 黑盒变白盒 · 审动作 · 对证据 · 查时序

> **一句话**：Agent 说"做完了"——它到底做了什么？FindAction 从执行证据中还原动作序列，标出跳过的步骤、异常的行为、未声明的决策。
>
> **三节点**：① 采（收集执行证据）→ ② 审（还原动作 + 比对计划）→ ③ 出结果（动作清单输出）。多一步就是多余。

---

## 触发条件

用户说 `/findaction` `/fa` `审执行` `审动作` `action 一下`。

**适合**：Agent 执行了一组操作后，你想知道它到底做了什么。
**不适合**：审内容对不对（那是 FindMiss 的事）、审 prompt 清不清楚（那是 FindGap 的事）。

---

## 与 FindGap / FindMiss 的分工

```
FindGap    事前    你的请求缺了什么（照输入）
FindAction 事中    Agent 做了什么  （照过程）  ← 你在这里
FindMiss   事后    结果漏了什么    （照输出）
```

FindAction 只回答一个问题：**Agent 在执行过程中，做了哪些动作，跳过了哪些步骤，哪里有异常？**

不判断内容对错，不评估结果好坏。做了就是做了，没做就是没做。

---

## 内部审查引擎（7 条规则，不对外展示代号）

以下规则在内部驱动执行审查，但输出中**只用人话描述发现，不展示规则代号**。

| 内部代号 | 审什么 |
|---------|--------|
| A-STEP | 计划/workflow 中声明的步骤是否全部有执行证据 |
| A-SKIP | 是否有步骤被跳过但未说明原因 |
| A-ORDER | 有依赖关系的步骤是否按正确顺序执行 |
| A-RETRY | 是否有步骤失败后重试，重试了几次，最终是否成功 |
| A-SILENT | 是否有步骤执行失败但被静默跳过（未报错继续下一步）|
| A-SCOPE | 实际修改/操作的范围是否超出预期（改了不该改的文件、访问了不该访问的资源）|
| A-DECIDE | Agent 在执行中做了哪些分支决策（选了 A 方案放弃了 B 方案）但未向用户说明 |

### 规则边界

- A-STEP vs A-SKIP：STEP 检查有没有证据，SKIP 检查跳过有没有理由
- A-RETRY vs A-SILENT：RETRY 是显式重试（有记录），SILENT 是失败被吞（无记录但后续行为暗示跳过）
- A-SCOPE：只检查操作范围，不判断操作内容对错
- A-DECIDE：只暴露决策点，不评判决策好坏

## 三节点执行流程

### 节点 ① 采（收集执行证据）

从以下来源收集 Agent 的执行证据：

| 证据来源 | 采集方式 | 证据强度 |
|---------|---------|---------|
| 工具调用记录 | 读取对话中的 tool_use / tool_result | 强证据 |
| 文件变更 | `git diff` / `git status` | 强证据 |
| 命令执行日志 | Bash 工具的输入输出 | 强证据 |
| Agent 的文本声明 | "我已经完成了 X" | 弱证据（self-report） |
| 无记录 | 找不到任何执行痕迹 | 默认未执行 |

**关键原则**：
- self-report 不算 proof——Agent 说"做了"不等于真做了
- 缺证据默认未执行——宁可误判"没做"，不放过"假装做了"

### 节点 ② 审（还原动作 + 标注异常）

**步骤 2.1：还原动作序列**

从证据中提取 Agent 的实际执行路径，按时间顺序排列：

```
步骤 1: 读取了 src/auth/ 下的 4 个文件
步骤 2: 修改了 src/auth/middleware.ts（+23行 -8行）
步骤 3: 修改了 src/auth/types.ts（+5行 -2行）
步骤 4: 运行了 npm test（失败，2 个用例未通过）
步骤 5: 修改了 src/auth/middleware.ts（+3行 -1行）
步骤 6: 运行了 npm test（通过）
```

**步骤 2.2：比对计划**

如果用户给出了明确的计划/workflow/步骤列表，逐条比对：

| 计划步骤 | 执行证据 | 状态 |
|---------|---------|------|
| 修改 auth 中间件 | Edit middleware.ts ×2 | ✅ 已执行 |
| 添加单元测试 | 无相关文件变更 | ❌ 未执行 |
| 更新 README | 无相关文件变更 | ❌ 未执行 |

如果用户没有给出明确计划，则只输出动作序列 + 异常标注，不做比对。

**步骤 2.3：标注异常**

对每个异常行为标注类型和严重度：

| 异常类型 | 严重度标记 | 说明 |
|---------|-----------|------|
| 步骤跳过无理由 | 🔴 需关注 | 计划中有但找不到执行证据 |
| 静默失败 | 🔴 需关注 | 某步失败但未报错继续执行 |
| 范围越界 | 🟠 请留意 | 改了计划外的文件/资源 |
| 未声明决策 | 🟠 请留意 | Agent 做了选择但没告诉你 |
| 多次重试 | 🟡 可忽略 | 某步失败后重试 ≥2 次 |
| 顺序偏差 | 🟡 可忽略 | 执行顺序与计划不同但结果不受影响 |

### 节点 ③ 出结果（动作清单输出）

**输出模板**：

```
FindAction · 还原 N 个动作，发现 M 处异常

---

📋 执行路径

步骤 1: {动作描述}
步骤 2: {动作描述}
  └─ 🟠 {异常标注}
步骤 3: {动作描述}
...

---

🔴 需关注 · {异常名称}

证据：{具体的执行记录或缺失记录}
影响：{这个异常可能导致什么后果}

---

🟠 请留意 · {异常名称}

证据：{具体的执行记录}
说明：{Agent 做了什么未声明的决定}

---

（如有计划比对）

📊 计划完成度

| 计划步骤 | 状态 |
|---------|------|
| {步骤1} | ✅ 已执行 |
| {步骤2} | ❌ 未执行 |
| {步骤3} | ✅ 已执行 |

完成 X/Y（Z%）
```

**输出规则**：
- 🔴 必须给"证据"和"影响"
- 🟠 必须给"证据"和"说明"
- 🟡 只需简短标注
- 每条异常必须附可追溯的证据来源
- 不输出任何规则代号
- 不评判动作的对错，只陈述事实

---

## 红线（4 条）

1. **只审执行过程，不审内容对错**——改了什么是事实，改得对不对不归这里管
2. **缺证据默认未执行**——不脑补"它可能在线下做了"
3. **self-report ≠ proof**——Agent 声称完成不算完成证据
4. **不替用户判断**——只展示 Agent 做了什么，不说应该怎么做

---

## 杜绝（明确不做什么）

- ❌ 判断代码写得好不好（那是 code review）
- ❌ 判断结果对不对（那是 FindMiss 的事）
- ❌ 替用户决定是否接受执行结果
- ❌ 运行时维护静态知识库
- ❌ 输出中展示规则代号
- ❌ 给评分或 PASS/FAIL
